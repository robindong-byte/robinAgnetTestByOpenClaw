📖 理论部分（约 30 分钟）
为什么 Agent 需要记忆？
没有记忆的 Agent：
用户："我叫 Robin"
Agent："你好 Robin！"
---- 下次对话 ----
用户："我叫什么名字？"
Agent："我不知道..." ❌

有记忆的 Agent：
用户："我叫 Robin"
Agent：存入向量库
---- 下次对话 ----
用户："我叫什么名字？"
Agent：检索向量库 → "你叫 Robin！" ✅
传统数据库 vs 向量数据库
| 对比项 | 传统数据库（MySQL） | 向量数据库（ChromaDB） |
|--------|-------------------|----------------------|
| 存储内容 | 结构化数据 | 文本 + 向量（Embedding） |
| 查询方式 | 精确匹配（WHERE id=1） | 语义相似度搜索 |
| 适用场景 | 用户信息、订单 | 文档检索、记忆系统 |
| 查询示例 |   SELECT * WHERE name='Robin'   |   找和"狗"最相似的文本   |
ChromaDB 核心概念
ChromaDB
└── Collection（集合，类似数据库的表）
└── Document（文档）
├── id → 唯一标识
├── document → 原始文本
├── embedding → 文本的向量表示（自动生成）
└── metadata → 附加信息（时间、来源等）
什么是 Embedding？
文本  ──→  Embedding 模型  ──→  向量
"猫"  ──→  [0.2, 0.8, 0.1, ...]  （768维浮点数）
"猫咪" ─→  [0.21, 0.79, 0.11, ...]  （和"猫"很接近）
"汽车" ─→  [0.9, 0.1, 0.7, ...]   （和"猫"很远）

语义相似 → 向量距离近 → 能被检索到
